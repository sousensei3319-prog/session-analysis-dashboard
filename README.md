# Session Analysis Dashboard

時間帯（セッション）別の値動き傾向・トレード成績を可視化するStreamlitダッシュボード。
BTC/ETH/SOL/HYPE/GOLD/NQ/SP500/日経225 の1時間足を取得し、JST基準の9つの詳細時間帯（親4セッション）ごとに
騰落率・出来高・ボラティリティ・（アップロードした）トレード成績をクロス集計する。

「自分がどの時間帯で勝ちやすく／負けやすいか」を一目でわかるようにすることが目的（閲覧・分析専用。発注機能なし）。

Python 3.11系を推奨（開発・検証環境: Python 3.11.9）。

```bash
pip install streamlit yfinance ccxt plotly pandas numpy kaleido
```

検証済みバージョン: streamlit 1.59.1 / plotly 6.9.0 / kaleido 1.3.0 / ccxt 4.5.52 / yfinance 1.4.0 / pandas 3.0.1 / numpy 2.4.6

## 2. 起動方法

リポジトリルート（`C:\Users\kurim\OneDrive\デスクトップ\sou\Claudcode`）から:

```bash
streamlit run session_dashboard/app.py
```

ブラウザが自動で開かない場合は `http://localhost:8501` を開く。

### セルフテスト（ネットワーク不要・純ロジック検証のみ）

```bash
python session_dashboard/app.py --selftest
```

`SELFTEST: ALL PASS` が出力され exit code 0 で終了すれば正常（Streamlitサーバーは起動しない。`st.*` を一切呼ばずに全ロジックを検証する）。
検証項目（仕様書§7）:

1. hourマップ: 0〜23全hourが一意に1帯へ割当・9帯全て非空・大枠マップ整合
2. セッション境界判定（5:59/6:00等の境界値）
3. `compute_session_stats`: 既知値を仕込んだ合成OHLCVで手計算と一致
4. トレード集計: 回数/勝率/合計損益/プロフィットファクター/最大ドローダウン/最大連敗数
5. `normalize_trades_csv`: 列名ゆらぎ・Win_Loss欠落・壊れた行の吸収
6. `build_cross_table`: 市場統計のみ／トレードのみ／両方の3ケースで行列構造が正しい

## 3. 使い方（サイドバー）

1. **① 期間選択**: 開始日・終了日（デフォルト=今日から過去90日）。プリセットボタン（過去7日/30日/90日/1年）で一括変更。
   - yfinance の1時間足は**直近730日まで**しか取得できない制約があるため、開始日がそれより古い場合は自動的に729日前へクランプし、警告を表示する。
2. **② 銘柄選択**: BTC, ETH, SOL, HYPE, GOLD, NQ, SP500, 日経225 のチェックボックス（デフォルトONはBTCのみ）。
   「その他」欄にティッカーを入力し「追加」すると任意銘柄をチェックボックスとして追加できる（削除も可能）。
   ルーティング規則: `/` を含む文字列（例: `DOGE/USDT`）は CCXT(binance)、それ以外（例: `SPY`）は yfinance として扱う。
3. **③ 足種選択**: 1分足(暗号のみ) / 5分足 / 15分足 / 1時間足（デフォルト） / 4時間足 / 日足。
   **セッション集計は常に1時間足ベース**で行う（分足/4h/日足はチャート表示専用の見た目の変更のみ、集計は不変）。
   分足選択時の制約（実測に基づく。詳細は本README末尾「16. 追補v4 §2」参照）:
   - 1分足はccxt(暗号)銘柄専用。yfinance銘柄（GOLD/NQ/SP500/日経225等）は自動的に15分足へフォールバックし、
     その旨をcaptionで案内する。暗号銘柄を1つも選んでいない状態では1分足は選べない。
   - チャートのバー数破綻を防ぐため、表示期間が長すぎる場合は「1分=直近3日/5分=直近15日/15分=直近45日」に
     チャート表示窓だけ自動短縮する（セッション集計は選択期間全体のまま変わらない）。
4. **④ 比較モード**: トグルONで Period B 用の期間選択が追加表示され、A/Bの2期間を並べて比較できる。
5. **チャートオプション**: セッション背景帯 ON/OFF。期間が120日を超える場合は表示が煩雑になるため自動的にOFFになる（案内あり）。
   直下に**「背景帯の濃さ」スライダー**（5〜70%・5%刻み・既定25%）があり、背景帯トグルOFF時は操作不可（disabled）になる。
   詳細は本README末尾「13. 追補v3 §3」参照。
   さらにその下に**「トレードマーカー表示」トグル**（既定ON。トレードCSV読込時のみ実際に描画される）があり、
   複数データセット読込時は表示するデータセットを選ぶmultiselectが追加表示される。詳細は「16. 追補v4 §2」参照。

## 4. タブ構成

全タブ共通で、データ表示部の直下に**データ取得元キャプション**（📊 データ元/⏱ 取得時刻/最終足。詳細は
本README末尾「14. 追補v3 §4」参照）が表示される。

- **📊 多重クロス表**: 大枠4セッション（アジア/ロンドン/NY/薄商い）× 詳細9帯 のクロス表。
  平均騰落率・平均出来高・平均ボラティリティ・（トレードCSVがあれば）エントリー回数・勝率・合計損益・動きの特徴・Tips。
  **勝率<50%のセルは赤系、≥50%は緑系のグラデーションで色付け**され、負けやすい時間帯が一目でわかる。
  勝率<45%の帯はTipsに「⚠️ あなたはこの時間帯で負けやすい」が動的に追加される。CSVダウンロード可。
- **🕯️ チャート**: **🌊ボラチャート**（1段目・全選択銘柄を「期間先頭0%基準の累積騰落率(%)」で重畳表示。
  値幅の比較専用で価格そのものではない）＋**💹価格チャート**（2段目以降・銘柄ごとに1行、実価格の
  ローソク足を独自y軸で表示）の縦2段以上構成（単一銘柄選択時は末尾に出来高行も追加）。
  全行の時間軸は連動し、どの行にカーソルを置いても全行に同時刻の縦ラインが表示される。
  セッション背景帯（薄商い=グレー/アジア=青/ロンドン=緑/NY=赤）は全段を貫通して表示。
  PNGダウンロード（kaleido）、失敗時はHTMLダウンロードに自動フォールバック。
  トレードCSV読込時は▲LONG/▼SHORT/◆Side不明のエントリー＋✕決済マーカー、勝敗色の接続点線を
  **💹価格チャート行（実価格座標）にのみ**重畳表示（🌊ボラチャート行には描かない。既定ON、サイドバーでOFF可）。
  チャートクリックで時刻詳細を欄外パネルに固定表示するモード（既定）と、ホバーで日本語ツールチップを
  出すモードを切替可能。タブ下部には**🔍トレードズームビュー**（個別トレードの前後をオンデマンドで分足取得して精査）
  が出現する。詳細は本README末尾「16. 追補v4 §2」「17. 追補v5」参照。
- **📈 セッション分析**: 大枠4セッション別の平均騰落率・平均ボラの棒グラフ、銘柄間相関ヒートマップ（1h対数リターン）、
  セッション×銘柄の騰落率ヒートマップ。
- **📒 トレード履歴**: トレードCSVを**複数（ラベル付き）**アップロードすると自動でセッション判定・総合統計（総損益/取引数/勝率/PF/平均利益・損失/RR/最大DD/最大連敗）
  ・累積損益曲線・セッション別損益棒グラフを表示。テンプレートCSVのダウンロードや「サンプル(弟子+師匠)を読み込む」ボタンあり。
  データセットが2件以上あるときは各タブに「トレード統計の対象」選択欄が現れ、**👥 トレーダー比較ビュー**（2データセットの並列比較）も出現する。
  詳細は本README末尾「15. 追補v4 §1」参照。
  **アップロードしたトレード統計は多重クロス表・曜日/月別タブへ自動反映される**（トレードは選択期間でフィルタせず全件を反映。集計期間は脚注に別記）。
- **📅 曜日・月別**（追補§4。詳細は本README末尾「12. 追補§4」参照）: 上段=曜日別アノマリー（サイドバーの選択期間の1時間足をそのまま流用、追加フェッチなし）、
  下段=月別アノマリー（選択期間とは**独立**に長期日足を別途取得。lookbackで3年/5年/10年を選択）。
  いずれもトレードCSVをアップロード済みなら曜日別・月別の勝率ヒートマップ／サマリーが自動反映される。
- **⚡ 確率スキャナー**（追補v6。詳細は本README末尾「18. 追補v6」参照）: サイドバーの期間選択・銘柄選択とは
  **独立**に動作する（BTC/ETH/SOL専用・`data_1m/`の1分足を`scan_job.py`が事前集計した`scan_results.json`を読む）。
  月×曜日×30分刻みの時間帯別に騰落方向・値幅の統計的傾向（n/Wilson信頼区間/BH-FDR補正/前後半安定性/
  手数料込み期待値）を提示する。ランキング／1日マップ／セル詳細の3ビュー構成。

## 5. トレード履歴CSVフォーマット仕様

`sample_trades.csv`（弟子側・このディレクトリに同梱、30件・全9詳細帯に散らばり・複数銘柄・勝ち負け混在）と
`sample_trades_mentor.csv`（師匠側・同期間/同銘柄群で成績が明確に良い合成データ30件）を参照。

### 必須列

| 列名 | 内容 |
|---|---|
| `Entry_Time` (または `Entry_Time(JST)` 等) | エントリー日時（JST、`YYYY-MM-DD HH:MM:SS`） |
| `Symbol` | 銘柄（例: BTC, ETH, SOL, HYPE, GOLD ...） |
| `PnL_USD` | 損益（USD、数値。`"1,234.5"` や `"$100"` のような文字列表記も自動でクレンジングされる） |

### 任意列

| 列名 | 内容 |
|---|---|
| `Exit_Time` (または `Exit_Time(JST)` 等) | 決済日時（JST） |
| `PnL_Percent` | 損益率（%） |
| `Win_Loss` | `win`/`loss`（日本語「勝ち」「負け」等も可）。**無ければ `PnL_USD` の符号から自動導出**（正=win、負=loss、`0`は勝敗どちらでもない**引き分け**として扱い、勝率の分母から除外する） |
| `Side` | `LONG`/`SHORT`（日本語「ロング/ショート」「買/売」等の表記ゆれも吸収。認識不能な値はNaN=Side不明扱い） |
| `Leverage` | レバレッジ倍率（数値。`"10x"`/`"x10"`のような文字列表記も自動で数値化。欠落/認識不能はNaN） |
| `Entry_Price` | エントリー価格（数値。`,`区切りや`$`/`¥`付きの文字列表記も自動でクレンジング。欠落時はチャート上でその時刻の足のcloseを近似値として使用し、hover/キャプションに「≈近似」と明記する） |
| `Exit_Price` | 決済価格（`Entry_Price`と同様の扱い） |

上記4列は追補v4で追加されたチャート上のトレードマーカー機能（「16. 追補v4 §2」参照）専用の任意列で、
無くても既存の全機能（クロス表・トレード統計・比較ビュー等）はそのまま動作する（後方互換）。

### 列名ゆらぎ吸収

`Entry_Time(JST)` / `entry_time` / `エントリー時刻` のような表記ゆれは自動的に正規化される。英語・日本語どちらのヘッダーでも読み込み可能。

### エンコーディング

`utf-8-sig` → `cp932` の順で自動判定して読み込む。

### 壊れた行の扱い

日時が解析できない・Symbolが空・PnLが数値変換できない・勝敗が判定できない行は**具体的な理由付き**でエラー表示された上でスキップされる（他の行は正常に読み込まれる）。

## 6. データソース（研究班R1実証済み・推測でティッカーを変更しないこと）

| 表示名 | 取得元 | ティッカー | 備考 |
|---|---|---|---|
| BTC | ccxt (binance) | `BTC/USDT` | 90日ページネーション2160本=100%網羅を実証済み |
| ETH | ccxt (binance) | `ETH/USDT` | |
| SOL | ccxt (binance) | `SOL/USDT` | |
| HYPE | ccxt (**bybit**) | `HYPE/USDT` | binanceはHYPE/USDT非上場（BadSymbol）のためbybitを採用 |
| GOLD | yfinance | `GC=F` | Comex金先物。90日で1409本、volume非ゼロ率96.4%、23/24hカバー |
| NQ | yfinance | `NQ=F` | Nasdaq100(E-mini)先物 |
| SP500 | yfinance | `ES=F` | S&P500(E-mini)先物。`^GSPC`は日中セッションのみ(7/24h)のため不採用 |
| 日経225 | yfinance | `NIY=F` | シカゴ日経225(円建)先物。`^N225`はvolume列が全ゼロのため不採用 |

- ccxt: `fetch_ohlcv(symbol, '1h', since=..., limit=1000)` をループしてページネーション、取得結果は `Asia/Tokyo` に変換して保持。
- yfinance: `yf.download(ticker, start, end, interval='1h', auto_adjust=False)`。1時間足は**直近730日制限**あり（サイドバーで自動クランプ）。
- 銘柄ごとに取得を try/except し、失敗した銘柄のみ警告表示して処理を継続する（全滅時はエラー表示）。
- `@st.cache_data(ttl=900)` で15分キャッシュ。発注・取引所への書き込みAPIは一切実装していない（閲覧専用、APIキー不要の公開エンドポイントのみ）。
- **分足（チャートタブ専用、追補v4§2.4A）**: ccxtは1分/5分/15分足とも対応（1分足は数年遡及可能、実測済み）。
  yfinanceは1分足を使わず、5分足/15分足のみ対応（実測に基づく制約。詳細は「16. 追補v4 §2」参照）。
  分足のキャッシュTTLは1分=5分・5分/15分=10分（1時間足系の15分より短い＝より新鮮なデータを優先）。

## 7. セッション定義（JST・§2.1 一意hourマップ、集計の正本）

参考にした一般的なセッション区分には時間帯の重複（16時台・21時〜2時）があるため、**集計は下表の一意マップで行い二重計上しない**（表示ラベルは重複込みの慣用表記のまま）。

| 詳細帯ラベル（表示） | 集計対象hour(JST) | 親（大枠セッション） |
|---|---|---|
| アジア早朝 (7-10) | 7, 8, 9 | アジア時間 |
| アジア本番 (10-14) | 10, 11, 12, 13 | アジア時間 |
| アジア後半 (14-17) | 14, 15 | アジア時間 |
| ロンドンオープン (16-18) | 16, 17 | ロンドン時間 |
| ロンドン中盤 (18-21) | 18, 19, 20 | ロンドン時間 |
| NY重複 (21-1) | 21, 22, 23, 0 | NY時間 |
| NY中盤 (1-4) | 1, 2, 3 | NY時間 |
| NY後半 (4-6) | 4, 5 | NY時間 |
| 薄商い (6-7) | 6 | 薄商いゾーン |

24時間を漏れなく・重複なくカバーする（selftestで検証済み）。大枠4行は詳細帯の時間数加重合成値。

## 8. PNGダウンロード（kaleido）について

「🕯️ チャート」タブの「📷 PNGを生成」ボタンは `fig.to_image(format="png", scale=2)` を実行する（1回あたり約2〜3秒）。
kaleidoはローカルのGoogle ChromeまたはMicrosoft Edgeを自動検出して利用するため、通常は追加設定不要。

もしPNG生成が失敗する場合（Chrome/Edgeが見つからない環境）は、以下のいずれかで対処できる。

1. Google ChromeまたはMicrosoft Edgeをインストールする。
2. コマンドラインで一度 `kaleido_get_chrome` を実行する（kaleido専用のヘッドレスChromiumを取得する）。
3. 上記が難しい場合は、アプリが自動的にHTMLダウンロードにフォールバックする。またはチャート右上のモードバーのカメラアイコンからも直接PNG保存できる。

## 12. 追補§4: 「📅 曜日・月別」タブ

サイドバーの期間選択・銘柄選択に連動する既存4タブ（多重クロス表／チャート／セッション分析／トレード履歴）に加え、
「いつ勝ちやすい／負けやすいか」を**曜日**単位・**月**単位でも見られるように追加したタブ。上下2セクション構成。

### 12.1 上段: 曜日別アノマリー（追補§1）

- **データ源**: サイドバーで選択中の期間の**1時間足をそのまま再利用**する（追加のネットワーク取得は発生しない）。
  そのため比較モード（Period A / B）にもそのまま対応し、A/B並列表示される。
- **集計単位**: JST暦日（0-23時のその日）を1オカレンスとし、月〜日の曜日ごとに平均する。
  セッション帯集計（多重クロス表タブ）の「NY重複」帯のような日またぎ繰り込みは行わない
  （23時台はその暦日、翌0時台は翌暦日の曜日にそのまま帰属する。意図した仕様であり境界バグではない）。
- **表示内容**: 曜日別サマリー表（平均騰落率／平均出来高／平均ボラティリティ、トレードCSVがあれば
  エントリー回数・勝率・合計損益も追加列で表示。勝率<50%は多重クロス表と同じ赤系グラデーション）、
  騰落率・ボラティリティの棒グラフ、セッション×曜日の騰落率ヒートマップ、
  （トレードCSVアップロード時のみ）セッション×曜日の勝率ヒートマップ。
- **銘柄セレクタ**: 「全選択銘柄の平均」または個別銘柄を選択できる（多重クロス表タブと同じUI）。

### 12.2 下段: 月別アノマリー（追補§2）— サイドバー期間から独立した長期日足

- **★最大の注意点: 月別集計だけはサイドバーの期間選択を使わない。**
  代わりにこのセクション内の **lookback セレクタ（3年 / 5年 / 10年、デフォルト5年）** で指定した年数分、
  銘柄ごとに**日足（'1d'）を別途取得**する（`fetch_daily_bundle`、1時間キャッシュ=ttl 3600秒）。
- **独立取得である理由**: サイドバーのデフォルト期間は過去90日程度であり、その中では「1月」「7月」のような
  暦月オカレンスが多くて1回・大半は0回にしかならず、月別平均は統計的に無意味（n=1以下）になる。
  月ごとの季節性を見るには最低でも数年分のオカレンスが必要なため、月別セクションだけ独立して
  長期の日足データを取得する設計にしている。この設計上の理由により、
  **月別アノマリーは比較モード（Period A / B）の対象外**（期間非依存の集計のため、A/B切替UIは表示されない）。
- **集計単位**: 暦月オカレンス（年×月の組）ごとに（月末close − 月初open）/月初open×100 を月次騰落率とし、
  1〜12月の月ごとに 平均／中央値騰落率・陽線率・平均ボラティリティ（日次対数リターンの母標準偏差）・
  オカレンス数（≒年数）を算出する。
- **表示内容**: 月別サマリー表（12行、トレードCSVがあれば月別エントリー回数・勝率・合計損益も追加列）、
  集計期間キャプション（lookbackで指定した年数分を**銘柄ごとに実際に取得できた**開始日〜終了日・本数を明記。
  取得元により実際の本数が銘柄間で微妙に異なりうるため、まとめてではなく銘柄ごとに脚注表示する）、
  月別騰落率の棒グラフ（銘柄別グループ棒）。

### 12.3 注意書き（掟2: 記述統計はエッジと呼ばない）

- 月別アノマリーはアプリ内にも警告文が表示される: 「n=年数程度しかなく統計的に弱い（多重検定でどれかの月は
  偶然目立つ）。参考情報であり単独でエッジと見なさない」。lookback=3年なら各月のオカレンス数は最大3、
  10年でも最大10であり、他タブ（多重クロス表・セッション分析）のような時間足ベースの大サンプルとは
  信頼性の桁が異なる点に注意。
- 曜日別アノマリーもトレードCSVのセル（特にn<3）は参考値である旨をキャプションで明示している
  （多重検定・生存バイアスの一般的注意はSPEC/CLAUDE.mdの鉄の掟2・3と同じ）。
- 月別集計はyfinance/ccxtへの**追加の日足リクエスト**を発生させる（1h集計とは別の取得）。
  lookbackを10年に設定して多数銘柄を選ぶと初回ロードがやや長くなる場合がある（1時間キャッシュされるため
  2回目以降は同一 (銘柄群, lookback) の組み合わせなら再取得は発生しない）。
- 曜日別・月別とも、選択期間・lookbackを変更すると銘柄別サマリー表・グラフ・ヒートマップは自動で再計算される
  （手動再読込み操作は不要）。

### 12.4 コードレビュー指摘対応（2026-07時点）

- **進行中の当月（欠け月）除外**: `fetch_daily_bundle`は`date.today()`まで日足を取得するため、実行時点を含む
  暦月は月末closeが確定しない「進行中の月」になる。`compute_month_stats`は呼び出し時点（`today_`引数）に
  一致する暦月オカレンスを平均／中央値騰落率・陽線率・n（年数）の集計から自動除外し、除外があった銘柄は
  「⚠️ 当月（YYYY年M月）は進行中（未完了）のため…除外しています」とキャプション表示する。
- **トレード集計期間の明記**: 曜日別・月別サマリー表のトレード由来列（エントリー回数・勝率・合計損益）は、
  表示中の市場データ期間（選択期間／lookback実取得期間）とは独立に**アップロード済み全トレード**から
  計算される。両セクションとも「トレード集計期間（全件・選択期間でフィルタしない）: 開始日〜終了日」を
  多重クロス表タブと同じ文言でキャプション表示する。
- **ヒートマップのデータ欠損時表示**: 曜日別セクションのセッション×曜日ヒートマップで、選択銘柄の取得が
  失敗している場合は「全選択銘柄の平均」へ無警告ですり替えず、「『(銘柄)』のデータが取得できなかったため…
  表示できません」と明示する（上のサマリー表がNaN/「—」表示になるのと整合させるため）。
- **n(日数)/n(年数)の表示丸め**: 「全選択銘柄の平均」選択時は複数銘柄のnを平均するため小数になりうる。
  従来の`int()`切り捨てでは実際のサンプル数を過小表示していたため、四捨五入（`round()`）表示に変更した。

## 13. 追補v3 §3: 「背景帯の濃さ」スライダー

サイドバー「④ チャートオプション」の「セッション背景帯」トグル直下に、背景帯の不透明度を調整する
スライダーを追加している。

- **範囲**: 5〜70%（5%刻み）。既定値は25%（従来のハードコード値0.10より視認性を上げたデフォルト）。
- **トグルとの連動**: 「セッション背景帯」トグルがOFFのときはスライダー自体が操作不可（`disabled`表示）になる。
  背景帯を表示しない設定で濃さだけ変えても意味がないため。
- **配線**: スライダー値（%）を100で割った小数（0.05〜0.70）が `build_session_background_shapes()` の
  `opacity` 引数へそのまま渡される。従来定数 `BG_OPACITY` はセルフテスト用のデフォルト引数としてのみ残し、
  UI描画時は必ずスライダー値で上書きする（ハードコード撤廃）。
- **再取得は発生しない**: 濃さスライダーは `fetch_ccxt_ohlcv` / `fetch_yfinance_ohlcv` / `compute_bundle` /
  `fetch_daily_bundle` の `@st.cache_data` キャッシュキー（銘柄・期間・lookback等）に一切含まれていない。
  スライダー操作でStreamlitのスクリプト全体は再実行されるが、上記キャッシュ関数は同一引数でヒットし
  ネットワーク再取得・重い再計算は発生しない（Plotly図のshape再構築のみ）。

## 14. 追補v3 §4: データ取得元キャプション（全タブ共通）

各タブのデータ表示部の直下に、実際にどこから・いつ取得したデータかを示すキャプションを表示する
（「見た目は最新だが実は古いキャッシュ」を利用者が判別できるようにするための機能）。

### 表示フォーマット（市場データ系タブ）

```
📊 データ元: <銘柄名>=<取得元>(<ティッカー>) | ⏱ 取得: YYYY-MM-DD HH:MM JST時点(キャッシュ<TTL分>分) | 最終足: MM-DD HH:MM JST
```

- **取得元**: `ccxt(binance)` / `ccxt(bybit)` / `yfinance` のいずれか（銘柄ごとの実ルーティング結果）。
- **⏱ 取得時刻**: `fetch_ccxt_ohlcv` / `fetch_yfinance_ohlcv` が実際にフェッチを実行した瞬間（`@st.cache_data`の
  キャッシュmiss時のみ）のJST時刻。**キャッシュhit時は再フェッチされないため、画面描画時の
  `datetime.now()` で代用することは禁止**（見かけ上「今取得した」ように見せる嘘表示になるため）。
  この時刻は取得結果のタプル `(df, meta)` の `meta["fetched_at"]` にフェッチ関数内部で記録され、
  st.cache_dataのメモ化によってキャッシュが効いている間はそのまま保持される。
- **キャッシュTTL**: 1時間足系（多重クロス表／チャート／セッション分析／曜日別）は15分
  （`HOURLY_CACHE_TTL_SEC`）、月別セクションの長期日足は1時間（`DAILY_CACHE_TTL_SEC`）。
- **最終足**: 取得したOHLCVの最終行のタイムスタンプ（JST）。「このデータは何時の足まで含んでいるか」を示す。

### タブ別の表示箇所

| タブ | 表示内容 |
|---|---|
| 📊 多重クロス表 | 選択中銘柄（「全選択銘柄の平均」なら寄与銘柄すべて）を列挙。比較モードはA/B両方のパネルに表示。 |
| 🕯️ チャート | 表示中銘柄ごとに1行。 |
| 📈 セッション分析 | 選択中銘柄（平均選択時は寄与銘柄すべて）。比較モードはA/B両方。 |
| 📅 曜日・月別（上段=曜日別） | 曜日別集計に使う1時間足のキャプション（サイドバー期間の使い回しのため多重クロス表と同じ`fetched_at`になる）。比較モードA/B対応。 |
| 📅 曜日・月別（下段=月別） | 月別集計専用に別取得した**日足**のキャプションを、選択期間側（1h）のキャプションとは別行で表示（取得元・TTLとも異なるため）。比較モード非対応（月別自体が期間非依存のため）。 |
| 📒 トレード履歴 | 市場データではなくトレードCSVの読込元。「📊 データ元: `<アップロードファイル名>` または `サンプルデータ(アプリ内生成)` \| ⏱ 読込: YYYY-MM-DD HH:MM JST時点」。読込時刻はアップロード/サンプルボタン押下の瞬間に記録（描画時の`now()`代用ではない）。 |

### 純関数化

キャプション文字列の整形自体は `format_data_source_caption(label, meta, cache_ttl_min)` という
`st.*` に依存しない純関数で行っている（テスト容易性のため。ロジック本体はメタ辞書の欠落・不明値の
フォールバック処理のみで、フェッチ層の`fetched_at`をそのまま文字列化するだけであり日時演算を行わない
ため、セルフテストは主にフェッチ層側 `_build_fetch_meta` の契約（`source`/`ticker`/`fetched_at`/`last_bar`/`rows`
の5キーが必ず揃うこと）で担保している）。

## 15. 追補v4 §1: 複数トレードCSVラベル管理 + 👥 トレーダー比較ビュー

目的: CSV1=弟子（自分）・CSV2=師匠のように**複数のトレード履歴を並べて比較**し、
「自分のトレードを師匠に近づけるにはどうすればいいか」をデータで確認できるようにする。

### 15.1 読み込みUI（📒 トレード履歴タブ）

- `st.file_uploader` は複数ファイル選択（`accept_multiple_files=True`）に対応。アップロードごとに
  **編集可能なラベル欄**（既定値=ファイル名から拡張子を除いたもの）が表示され、各データセットの
  削除ボタンも独立して用意されている。
- 同名ラベルは自動で連番が付与される（`ラベル` → `ラベル_2` → `ラベル_3` ...）。ラベル入力欄に
  重複を入力した場合、入力欄自体の表示はStreamlitの仕様上そのまま残るが、直下に
  「→ 実際のラベル: `...`（重複する場合は自動で連番付与）」という注記で実際に使われるラベルを明示する。
- 「サンプル(弟子+師匠)を読み込む」ボタンは、既存の弟子サンプル（`sample_trades.csv`）と
  新設の師匠サンプル（`sample_trades_mentor.csv`。同期間・同銘柄群で成績が明確に良い合成データ）を
  **既存データセットを消さずに追記**する（クリックするたびに新規2件が追加される）。
- **後方互換**: アップロードが1件（またはサンプル未読込で0件）のときはラベル管理UI・選択欄は
  一切表示されず、追補前と全く同じ単一データセット表示になる。

### 15.2 各タブでのデータセット選択（📊 多重クロス表 / 📅 曜日・月別 / 📒 トレード履歴）

- データセットが**2件以上**あるとき、トレード統計を使う3タブそれぞれに独立した
  「トレード統計の対象」selectbox（既定=先頭のラベル）が表示され、選んだデータセットの
  トレード統計のみがそのタブに反映される（タブ間で選択は連動しない＝タブごとに別のデータセットを見比べ可能）。
- データセットが1件のときはselectbox自体が非表示になり、追補前と同じ挙動になる。

### 15.3 👥 トレーダー比較ビュー（📒 トレード履歴タブ内・データセット2件以上で出現）

比較したい2つのデータセットをselectboxで選ぶ（既定=先頭の2件）と、以下が表示される。

1. **総合統計の並列表**: 総損益／取引数／勝率／PF／平均利益／平均損失／RR／最大DD／最大連敗を
   2データセット分並べた表。
2. **累積損益曲線の重ね描き**: 2本を色分けして同一グラフに重畳。PnLの規模が異なる場合に備え
   「1トレードあたり平均PnLで正規化」トグル（既定OFF=生USD表示）。
3. **詳細帯別 比較表**: 詳細9帯 × [各データセットの回数／勝率 + 勝率差]。勝率差は発散色付け
   （比較対象2側が優位=赤系＝改善余地、比較対象1側が同等以上=緑系）。該当帯のトレード数が3件未満の
   側には「(n小)」注記が付く。
4. **改善ヒントリスト（数字のみ・一般論のアドバイス文は生成しない＝掟2）**: 「相手との勝率差が
   大きい帯TOP3」（差・両者のn・両者の勝率を明記）と「エントリーが多い（n≥3）のに勝率が
   `LOW_WINRATE_THRESHOLD`（45%）未満の帯」を、比較対象1側の視点でリスト表示する。
5. **曜日別 勝率差ミニ表**（7行。③と同じ形式・色付け）。
- 全ての比較出力に**両データセットの集計期間（最初のトレード〜最後のトレード）を明記**する（掟6）。

### 15.4 純ロジック

- `compare_trade_stats(stats_a, stats_b)`: `compute_trade_stats()` の出力2つを受け取り、
  総合統計の差分・帯別（または曜日別）の並列＋勝率差・低n判定・両者の集計期間を返す純関数
  （`st.*` 非依存・selftest対象）。行の対象は呼び出し時に渡した DataFrame の index を
  そのまま使うため、帯別9行にも曜日別7行にも同じ関数を流用できる。
- UI層のデータセット管理（`_add_trade_dataset_record` / `_dedupe_trade_label` /
  `_load_sample_trade_datasets` / `_render_trader_comparison_view` 等）は `render_*` 系関数と同じく
  `st.*` 呼び出しを含むためselftest対象外（既存の設計方針どおり、UI層とロジック層を分離）。

## 16. 追補v4 §2: チャート上のトレードマーカー + 🔍トレードズームビュー + 分足対応

目的: 「どこで・何倍レバで・どこで決済したか」をチャート上で直接確認できるようにし、個別トレードを
分足レベルまで掘り下げて検証できるようにする。

### 16.1 トレードマーカー（🕯️ チャートタブ）

- サイドバーの「トレードマーカー表示」トグル（既定ON）で有効化。トレードCSV未読込時は何も描画されない。
- 複数データセット読込時はマーカー表示データセットを選ぶmultiselect（既定=全データセット）が追加表示され、
  データセットごとに輪郭色を変える（弟子=水色系`#5DE0E6`/師匠=金色系`#FFD24C`、以降は6色サイクル）。
  各データセットは凡例に「<ラベル> エントリー」「<ラベル> 決済」として登録される。
- 描画対象は「チャート表示中の銘柄 × 表示期間内」のトレードのみ。銘柄の一致判定はSymbol列とチャート銘柄の
  表示ラベル/実ティッカーを英数字のみ・大文字に正規化した上で完全一致または部分一致で照合する
  （一致しない銘柄のトレードは描かれない。ルールはチャート直下のcaptionに明記）。
- **描画先（追補v5§4で確定）**: マーカーは常に**該当銘柄の💹価格チャート行（実価格座標）**に描画する。
  🌊ボラチャート行（累積騰落率%）には描かない（視認性優先。selftest §17で「全トレードtraceがボラ行には
  無い」ことを検証済み）。
- **エントリー**: ▲(LONG)/▼(SHORT)/◆(Side不明)。y座標=Entry_Price（あれば実価格、無ければその時刻の足の
  close値を近似として使用し、hoverに「≈近似(足のclose)」と明記）。
- **決済**: ✕マーカー（Exit_Priceも同様のフォールバック）。Exit_Time欠落トレードはエントリーのみ描画。
- エントリー→決済を点線で接続（勝ち=緑`#2ECC71`/負け=赤`#E74C3C`/引分・不明=グレー`#AAAAAA`）。
- **hover表示**: データセットラベル・Side・レバレッジ（欠落時「—」）・PnL(USD/%)・保有時間・価格が
  実価格か近似かの別。
- **マーカーサイズ**: レバレッジ段階スケール（`leverage_marker_size()`。目安: 〜2x=小、〜10x=中、
  〜25x=大、25x超=最大。欠落時は中間サイズ）。チャート直下のcaptionにスケールの説明を表示。
- 🔍トレードズームビュー（単一銘柄・実価格チャートのみの構成）でも同じ`add_trade_overlay_traces`を
  流用してマーカーを描く（`map_trades_to_chart`の`normalize_base`はズームビューでは常にNone=実価格）。
- 表示期間内に該当トレードが0件のときはチャートを邪魔せず、caption「表示期間内に該当トレードなし」のみ
  表示する。

### 16.2 🔍 トレードズームビュー（チャートタブ下部）

- トレードCSV読込時のみチャートタブの下部に出現。データセットが複数ある場合はまずデータセットを選び、
  次にそのデータセット内のトレードをselectboxで選ぶ（表示例:
  「#12 BTC LONG 10x 07-03 14:23→15:41 +120.5USD」）。
- 選択すると、そのトレードの**エントリー2時間前〜決済（無ければエントリー）2時間後**の窓だけを
  オンデマンドで取得する（`compute_zoom_window`）。銘柄はトレードのSymbol列から解決するため、
  チャートタブで銘柄を選択していなくても単独で動作する。
- **足種の自動選択**（`resolve_zoom_timeframe`、ソース別・実測に基づく）:
  - 暗号(ccxt): 窓≤12時間→1分足 / ≤3日→5分足 / ≤10日→15分足 / それ以上→1時間足。
  - yfinance（1分足は使わない方針）: 窓≤3日かつトレードが直近60日以内→5分足（60日超は15分足へ
    フォールバック+理由表示）/ 窓≤10日→15分足 / それ以上→1時間足。
  - 自動選択は「足種(自動を手動上書き可)」selectboxで手動上書きできる。
  - 取得に失敗した場合（上場前・yfinanceの遡及限界超え等）は、より粗い足種へ段階的に自動フォールバック
    し、その旨をcaptionで表示する（`fetch_zoom_window_ohlcv`のフォールバック梯子）。
- 描画: ローソク足＋エントリー▲/決済✕マーカー（実価格 or ≈近似・§16.1と同じ作法）＋エントリー→決済の
  点線＋**エントリー価格/決済価格の水平点線**＋セッション背景帯（サイドバーの「背景帯の濃さ」スライダーを
  共有）。hoverはSide/レバ/PnL/保有時間。データ取得元キャプション（§14）も表示される。

### 16.3 分足対応（🕯️ チャートタブの足種選択・追補v4§2.4A）

- サイドバー③足種選択に「1分足(暗号のみ)」「5分足」「15分足」が追加され、既存の1時間足(既定)/4時間足/
  日足と合わせて6択になった（`TIMEFRAME_CHOICES`）。**セッション集計（クロス表・セッション分析・曜日/月別）
  は常に1時間足ベースで選択期間全体を対象にする**ため、分足選択はチャート表示の見た目にのみ影響する。
- 1分足はccxt(暗号)銘柄専用。yfinance銘柄（GOLD/NQ/SP500/日経225等）で1分足を選ぶと自動的に15分足へ
  フォールバックし、「GOLD等は1分足非対応(15分で表示)」とcaptionで明示する
  （`resolve_intraday_effective_choice`）。暗号銘柄を1つも選んでいない状態では1分足は選べない。
- **表示期間ガード**（バー数上限≈5,000本目安。`compute_intraday_chart_window`）: 1分足=直近3日以内・
  5分足=直近15日以内・15分足=直近45日以内。選択期間がこれを超える場合は「チャートは直近N日のみ分足表示
  (セッション集計は従来どおり1時間足で全期間)」と警告した上で**チャート表示窓だけ**自動短縮する
  （集計ロジックは無関係のため不変）。
- 実測済みの制約（推測で決めていない。§6のデータソース節も参照）:
  - ccxt(Binance/Bybit): 1分足は数年単位で遡及可能。90日分=約13万本はPlotly描画が破綻するため
    上記の表示期間ガードで防止している。
  - yfinance: 5分足は概ね直近60日まで、15分足は730日まで が実用的な遡及限界（🔍ズームビューの
    `ZOOM_YF_5M_AGE_LIMIT_DAYS`定数に反映）。1分足はyfinanceでは使わない方針（ユーザー指示・2026-07-10）。

### 16.4 純ロジック

- `map_trades_to_chart(trades_df, ohlcv_df, normalize_base)`: トレードCSVの各行をチャート座標
  （entry_time/entry_price/exit_time/exit_price等）へ変換する純関数。価格フォールバック（近似close）・
  正規化変換・期間外除外をすべて担う（selftest対象。実価格あり/なし・正規化あり/なし・期間外除外・
  Exit欠落の各既知解を検証済み）。

  selftest対象外）。
- `compute_zoom_window(entry_time, exit_time)` / `resolve_zoom_timeframe(source, window_hours,
  trade_age_days)` / `format_trade_select_label(...)` / `build_zoom_trade_options(...)`: 🔍ズームビューの
  窓算出・足種自動選択・selectbox表示文字列組み立てをそれぞれ純関数化しselftest対象にしている。

## 17. 追補v5: 🌊ボラチャート＋💹価格チャート2段構成＋欄外詳細パネル

背景（ユーザー指摘・2026-07-10）: v4までの複数銘柄チャートは「先頭終値=100」の正規化指数であり、
価格そのものではなかった。意図はボラティリティ（値幅%）の比較だったため、それを明確にした
「🌊ボラチャート」に改称・再設計し、実価格の「💹価格チャート」を別行で同期表示するようにした。

### 17.1 チャートタブの行構成

- `make_subplots(rows=1+選択銘柄数(+単一銘柄時は出来高1行), shared_xaxes=True)` で1つのfigureに統合。
  - **1段目=🌊ボラチャート**: 全銘柄を重畳表示。値は「期間先頭終値からの累積騰落率(%)」
    （`cumulative_return_pct` = `(price/期間先頭close - 1) * 100`。先頭バーは常に0%）。y軸タイトル
    「累積騰落率(%)」。ローソク足のまま（OHLC4値をすべて累積騰落率に変換 = `to_cumulative_return_ohlc`）。
  - **2段目以降=💹価格チャート**: 銘柄ごとに1行、実価格のローソク足（行ごとに独自y軸 `y2`, `y3`, ...。
    BTCと金のような桁違いの銘柄を同じ軸に重ねない）。行タイトルに銘柄ラベルを注記。
  - 出来高行は**単一銘柄選択時のみ**末尾に維持（複数銘柄時は出来高行なし）。
  - 高さ = 1行あたり320px（行数に応じて縦長になる。縦スクロール前提でユーザー了承済み）。
- セッション背景帯は `yref='paper'` の1組のshapeで全段を貫通させる（行ごとに重複追加しない。
  行数を増やしても`len(shapes)`は変わらないことをselftest §15-4で検証）。

### 17.2 時間軸同期＋消えない縦ライン

- `shared_xaxes=True` で全行の時間軸（ズーム・パンも含む）を完全同期。
- 全xaxisに `showspikes=True, spikemode='across', spikesnap='cursor', spikethickness=1,
  spikedash='dot'` を設定（`fig.update_xaxes(...)` の一括適用。行ごとに個別設定すると罠⑤で
  一部の行だけ縦ラインが出ない事故が起きるため、必ず全行一括で適用すること）。`hovermode='x'`。
  どの行にカーソルを置いても、全行の同時刻位置に縦ラインが表示される。
- rangeslider は従来どおり無効。

### 17.3 詳細表示方法（チャートタブ上部の `st.radio`）

- **「欄外パネル（クリックで固定）」（既定）**: チャート内のホバーボックスは出さない
  （ローソク足/出来高trace は `hoverinfo='skip'`。縦ラインの spikes は表示されたまま）。
  各行に**透明scatterオーバーレイ**（各バーのclose位置・`opacity=0.01`・`hoverinfo='skip'`）を敷き、
  `st.plotly_chart(fig, on_select="rerun", selection_mode="points")` のクリック選択からx（時刻）を
  取得し、チャート直下の専用パネルに固定表示する（`build_click_detail`が純ロジック本体）。
  内容は全て日本語: 日時(JST)/銘柄ごとの始値・高値・安値・終値・騰落率(%)・出来高・累積騰落率(%)、
  該当トレードがあればラベル/Side/レバ/PnLも併記。選択が無い間は案内文を表示。
- **「チャート内ホバー」**: ローソク足/出来高に日本語hovertemplateを設定（例:
  「始値 12,345.00<br>高値 …」、ボラ行は「累積騰落率 +1.23%」）。透明オーバーレイは追加しない。
  hoverlabelはフォント小＋半透明背景（`font_size=11, bgcolor='rgba(30,30,30,0.75)'`）で被りを軽減。
  英語表記（open/high/low/close等）は両モードとも画面に出さない。
- どちらのモードでも縦ライン（spikes）は常時有効。

## 18. 追補v6: ⚡確率スキャナータブ

目的（ユーザー依頼2026-07-11・仕様追補v6§0）: 「月×曜日×時刻の特定の数分間に価格がX%動く確率が高い箇所」を
BTC/ETH/SOL専用の1分足で統計的に洗い出し、短期売買の判断材料にする。サイドバーの期間・銘柄選択とは
**独立**に動作するタブで、`session_dashboard/`とは別プロセスの`scan_job.py`が事前計算した
`scan_results.json`を読み込んで表示するだけ（このタブ自体は発注機能もHTTP取得も一切行わない＝掟5）。

### 18.1 統計的誠実さのガード（最重要・掟2）

本プロジェクトの前科（曜日アノマリー12ヶ月検証でFDR補正通過ゼロ／横断BTで戦場選抜不可／1分スキャは
手数料壁で全滅）を踏まえ、以下のガードが**揃わない確率表示は「エッジ」と呼ばない**。タブ冒頭の
「📐 統計ガードについて」expanderに常設で解説文を表示する。

1. **n（サンプル数）と Wilson 95%信頼区間**を全セルに常時表示。n<30は目安として注記。
2. **BH-FDR補正**（銘柄内セル横断・二項検定 vs 全体基準率）→ q値<0.10のセルのみ既定表示
   （「FDR候補のみ」トグルで全表示に切替可）。
3. **前半/後半の方向一致（安定性）**を各セルに表示。
4. **手数料込み期待値**（往復0.10%控除後・設定可）を併記。
5. 「確率は過去の発生頻度であり将来の保証ではない」旨と多重検定の注意書きを常設。
   さらに「グロス/手数料比はレバレッジを変えても不変」というハイレバ算数警告カード（`st.error`）を
   タブ最上部に常設し、本タブがポジションサイズ・レバレッジの助言でないことを明記する。

### 18.2 データ層・階層グループ・統計エンジン（仕様§1-3・app.py内・純ロジック）

- データ元: `data_1m/{BTC,ETH,SOL}/{YEAR}.parquet`（`backfill_1m.py`がPM側で構築・増分更新）。
  `load_1m(label, years=None)`で読み込み、JST tz-aware indexへ変換。銘柄ごとに逐次処理し
  メモリを抑える（float32維持）。
- 階層: 1分足×5本=5分ブロック、5分ブロック×6本=30分グループ（JST 00:00起点・日またぎなし・1日48枠）。
  各30分グループは時刻範囲ラベルと**所属セッション**（既存の一意hourマップ§7を30分粒度に継承）を持つ。
  セルキーは `(月[1-12|全月合算], 曜日, 30分枠[0-47])`。5分粒度はセル詳細ビュー内の内訳表示に使用。
- 各オカレンスの騰落率 `g = (close_last − open_first)/open_first ×100`、判定はσ適応（そのセルの過去|g|
  分布の中央値×k、k=1.0既定）または固定閾値（±0.15%等）の切替式。吹っ飛び系はMFE_up/MFE_down
  （高値/安値までの最大逸脱率）。
- セル集計（`compute_scan_cells`）: n・平均/中央値騰落率・平均出来高・ボラ・P(上昇)/P(下降)/P(レンジ)＋
  Wilson CI・テール確率（MFE≥0.3/0.5/1.0%）＋CI・二項検定p値→BH-FDR q値・前半/後半P(上昇)と方向一致
  フラグ・手数料込み期待値。全て純関数でselftest対象（「18.5」参照）。

### 18.3 `scan_job.py` の運用（仕様§4・スタンドアロン実行）

`scan_job.py`はStreamlitと無関係に単独実行するバックグラウンドジョブで、`scan_results.json`を生成・
更新する。ダッシュボード側（app.py）はこのJSONを読むだけで、自分では1分足の再取得もスキャン再計算も
行わない（UIの応答性を保つため）。

```bash
python session_dashboard/scan_job.py --run     # 本番実行: 増分取得→3銘柄スキャン→JSON書出→カレンダー照合→Discord通知
python session_dashboard/scan_job.py --dry     # 通知を送らずに生成のみ試す（動作確認用）
python session_dashboard/scan_job.py --selftest # scan_job.py自体の純ロジックテスト（app.py --selftestとは別）
python session_dashboard/scan_job.py --no-fetch # 1分足の増分取得をスキップ（既存data_1m/のみで再計算）
python session_dashboard/scan_job.py --no-calendar # 経済指標カレンダー照合をスキップ
```

- **処理順**: ①`backfill_1m.py`相当の増分1分足取得 ②BTC/ETH/SOLを**銘柄ごとに逐次**スキャン再計算
  （`del`+`gc.collect()`でメモリ解放。Oracle VM の `MemoryMax=450M` 制約内で完走する設計）
  ③`scan_results.json`をアトミック書き出し（`updated_at`はJST・銘柄別セル配列＋`top_candidates`
  ＝FDR q<0.10かつn≥30を満たす上位候補）④経済指標カレンダー照合（`calendar_cache.json`に6hキャッシュ。
  取得失敗時は照合スキップ＝空フラグのまま本体は止めない）⑤Discord通知。
- **Discord webhook**: 環境変数 `DISCORD_WEBHOOK_SESSION_SCAN`（無ければ `DISCORD_WEBHOOK_NEW_TEST`へ
  フォールバック）。秘密情報はコード直書きせず`.env`経由（D-015準拠）。embed内容=更新日時＋銘柄別
  トップ3候補セル（n/CI/FDR q/指標フラグ付き）＋免責1行。
- **経済指標カレンダー**: `api.nasdaq.com/api/calendar/economicevents`（無料・Mozilla風UA必須）。
  セルの時刻窓±15分に高重要度指標が重なる場合にフラグを付与するのみで、「指標はボラティリティを
  拡大させるが方向は予測不可（検証済み知見）」という注記が必ず併記される。
- Oracle側のsystemdタイマー設定（1日4回想定）・funnel経由のJSON公開・Claude定期リサーチ
  （`scan_commentary.json`生成）は**PM側の運用作業**であり、本ワーカーのスコープ外（仕様§7）。
